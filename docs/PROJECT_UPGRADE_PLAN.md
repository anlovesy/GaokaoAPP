# 高考志愿填报推荐系统升级方案

## 1. 项目定位升级

### 1.1 推荐项目名称

智能升学决策系统 / 高考志愿填报推荐系统

相比“志愿填报工具”，`智能升学决策系统`更像一个完整产品，强调的不只是查分换学校，而是：

- 基于位次和历史录取数据做可解释推荐
- 对滑档风险进行量化评估
- 结合兴趣、城市、专业、就业偏好生成个性化方案
- 通过 AI 顾问完成二次追问和决策解释

### 1.2 业务背景

高考志愿填报的核心痛点不是“没有数据”，而是“信息无法被普通家庭高效消费”：

- 信息不对称：高校、专业、专业组、选科限制、调剂规则分散在不同来源
- 滑档风险高：很多考生只看分数，不看位次、趋势和招生波动
- 分数利用率低：只追求“能上”，没有兼顾专业、城市和长期就业
- 决策链条长：考生、家长、老师关注点不同，容易反复修改方案

因此，这个项目的本质不是“查学校”，而是“帮助用户在不确定条件下做升学决策”。

### 1.3 用户画像

#### 考生

- 关注点：能上什么、专业喜不喜欢、城市是否接受、未来就业如何
- 典型需求：生成一版冲稳保方案，快速理解每个志愿为什么这样排

#### 家长

- 关注点：稳定性、滑档风险、学费、城市安全、就业回报
- 典型需求：优先保本科、保学校、保城市，降低填报失误

#### 教师 / 志愿顾问

- 关注点：效率、批量服务能力、方案解释能力、历史方案可追溯
- 典型需求：基于标准化流程快速出方案，并通过 AI 辅助解释与复盘

### 1.4 简历里的产品定位表达

可以写成：

> 负责设计并实现一个面向考生、家长和升学顾问的智能升学决策系统，基于省控位次表、院校专业录取线、选科约束和风险偏好，生成冲稳保志愿方案，并通过 AI 顾问完成方案解释和二次调整。

## 2. 系统架构设计

### 2.1 架构目标

升级后的系统要同时满足三件事：

- 能跑：支持真实数据导入、推荐、追问、历史记录
- 能讲：架构分层清晰，面试时能说明“为什么这么拆”
- 能扩：后续可接更多省份、更多年份、更多推荐策略

### 2.3 模块划分

#### 1）数据层

建议拆为以下核心实体：

- `university`：院校基础信息，含院校层次、地区、办学性质
- `major`：专业基础信息，含学科门类、就业标签、选科要求
- `admission_line_fact`：历年院校-专业-省份-科类录取线、最低位次、计划数
- `province_score_rank_fact`：分数-位次映射表
- `subject_requirement`：专业组/专业选科限制
- `user_profile`：用户画像、偏好配置、历史输入
- `recommendation_result`：推荐结果快照，便于复盘
- `experiment_log`：A/B 实验分组和行为日志

#### 2）推荐层

推荐层建议分成 6 个子模块：

- `profile-normalizer`：标准化输入，补齐位次、校验选科
- `candidate-recall`：先召回满足基础约束的院校专业组合
- `probability-estimator`：计算录取概率和滑档风险
- `preference-scorer`：按专业、城市、就业、学费等偏好打分
- `strategy-builder`：按冲稳保策略生成最终志愿梯度
- `reason-generator`：输出推荐理由和风险提示

#### 3）服务层

- `recommendation-service`：接收用户画像，返回结构化志愿表
- `advisor-service`：结合推荐结果和上下文做 AI 追问回复
- `data-service`：提供年份、省份、分数段、院校检索接口
- `user-service`：登录、权限、历史记录、画像管理
- `experiment-service`：AB 分组、埋点、版本回溯

#### 4）展示层

- 学生端：首页、填报页、推荐结果页、AI 顾问页
- 家长端：重点展示风险说明、保底层、学费和就业信息
- 顾问端：多学生管理、方案导出、历史方案复盘
- 管理端：数据导入、实验开关、推荐配置、用户权限

## 3. 核心算法优化

### 3.1 当前算法的优点

从现有代码看，项目已经有不错的工程雏形：

- 支持根据分数反查位次
- 支持结合真实导入数据和内置规则推荐
- 支持冲稳保分层
- 支持结合兴趣、城市、院校标签、调剂偏好做加权
- 支持为推荐结果生成解释

这已经不是“纯 if-else Demo”，而是推荐系统的雏形。

### 3.2 升级版算法目标

升级后，算法要从“规则推荐”提升为“召回 + 打分 + 重排 + 风险控制”的工程化推荐链路。

### 3.3 输入 / 输出定义

#### 输入

- 基础信息：省份、年份、科类、分数、位次、选科
- 偏好信息：意向专业、意向城市、学校层次、是否接受调剂、学费预算
- 风险偏好：激进 / 均衡 / 保守
- 扩展特征：就业倾向、升学倾向、身体限制、特殊招生计划

#### 输出

- 冲 / 稳 / 保分层志愿列表
- 每个候选项的录取概率
- 每个候选项的风险标签
- 推荐理由与不确定性说明
- 整体方案滑档风险评估

### 3.4 推荐算法链路

```text
输入画像
  -> 数据标准化
  -> 约束过滤
  -> 候选集召回
  -> 位次匹配得分
  -> 概率模型估计
  -> 偏好得分
  -> 综合排序
  -> 冲稳保分桶
  -> 风险校准与解释输出
```

### 3.5 候选召回

第一步不是排序，而是先“过滤掉不可能项”，减少计算量并提高解释性。

召回规则建议包括：

- 省份、年份、科类匹配
- 选科要求匹配
- 批次匹配
- 学费预算过滤
- 特殊计划约束过滤
- 身体条件或专业限制过滤

经过召回后，候选规模通常从几千条降到几百条，便于后续排序和缓存。

### 3.6 位次匹配模型

高考志愿推荐里，位次比裸分更稳定，因此应把位次作为核心特征。

定义：

- `user_rank`：考生位次，数值越小越好
- `major_line_rank`：目标院校专业近年最低录取位次
- `rank_margin = major_line_rank - user_rank`

解释：

- `rank_margin > 0`：考生位次优于往年门槛，偏安全
- `rank_margin ≈ 0`：接近边界，偏稳
- `rank_margin < 0`：考生位次弱于往年门槛，偏冲

标准化后可以得到：

```text
rank_margin_rate = (major_line_rank - user_rank) / max(user_rank, 1)
```

这样做的原因：

- 不同分数段的“位次差”绝对值不可直接比较
- 用比例而不是纯差值，更适合跨分段建模

### 3.7 分数 / 位次区间概率模型

建议基于近 3 到 5 年数据，为每个“省份-科类-院校-专业”构建一个趋势门槛。

可使用加权历史位次：

```text
expected_rank_line =
0.50 * rank_line_t-1 +
0.30 * rank_line_t-2 +
0.20 * rank_line_t-3
```

再结合波动性：

```text
volatility = std(rank_line_t-1, rank_line_t-2, rank_line_t-3) / expected_rank_line
```

录取概率可用一个轻量逻辑回归思路表达：

```text
P(admit) = sigmoid(
  w1 * rank_margin_rate
  + w2 * trend_score
  + w3 * enrollment_score
  - w4 * volatility
  - w5 * adjustment_risk
)
```

其中：

- `trend_score`：近几年门槛变高还是变低
- `enrollment_score`：招生计划数越高，录取更稳定
- `volatility`：波动越大，不确定性越高
- `adjustment_risk`：不服从调剂、专业组冷热不均时风险更高

如果数据量还不足以训练正式模型，也可以先用“规则拟合概率”：

- `P >= 0.75`：保
- `0.45 <= P < 0.75`：稳
- `0.25 <= P < 0.45`：冲
- `P < 0.25`：默认不进入结果集

### 3.8 志愿梯度策略

志愿表不能只做 TopN 排序，必须显式做“冲稳保梯度控制”。

建议输出结构：

- 冲：2 到 3 个，负责拉高上限
- 稳：4 到 5 个，作为主力录取区间
- 保：3 到 5 个，确保不过度滑档

如果用户是低分段或边缘分段，可以自动加厚保底层：

- 冲：2
- 稳：4
- 保：5 到 6

这样设计的原因：

- 真正的填报不是“最高分项优先”，而是“风险组合优化”
- 面试时能体现你理解业务规则，而不是只会机器学习打分

### 3.9 偏好打分模型

位次只解决“能不能上”，偏好打分解决“上哪个更合适”。

建议构建用户偏好得分：

```text
preference_score =
0.35 * major_match
+ 0.20 * city_match
+ 0.15 * school_level_match
+ 0.15 * employment_match
+ 0.10 * tuition_match
+ 0.05 * postgraduate_match
```

其中：

- `major_match`：专业方向与兴趣标签相似度
- `city_match`：意向城市是否匹配
- `school_level_match`：985/211/双一流/地域院校偏好
- `employment_match`：就业稳定、考公友好、升学友好
- `tuition_match`：是否满足预算
- `postgraduate_match`：是否适合考研深造

### 3.10 综合排序模型

最终排序不是只看概率，也不是只看偏好。

建议综合得分：

```text
final_score =
0.45 * admit_score
+ 0.30 * preference_score
+ 0.15 * school_quality_score
+ 0.10 * major_value_score
```

如果后续要进一步升级，可以把这个加权模型替换为：

- Learning to Rank
- XGBoost 排序模型
- GBDT + Logistic 校准

但在项目初期，`加权评分 + 概率校准`已经足够工程化，也更容易讲清楚。

### 3.11 风险控制模型

风险控制是这个项目最容易拉开档次的地方。

建议增加两层风险：

#### 单志愿风险

- `school_risk = 1 - P(admit)`
- 输出风险标签：低风险 / 中风险 / 高风险

#### 整体方案滑档风险

可以用近似独立假设估算整体滑档概率：

```text
P(slip_plan) ≈ ∏(1 - P(admit_i))
```

再加上修正项：

- 如果多个志愿都来自同一分段、同一专业组，相关性更高，风险应上调
- 如果保底层不够厚，也应额外上调方案风险

最终可以输出：

- 当前方案整体安全度
- 是否存在“保底不足”
- 是否存在“城市偏好过强导致候选过窄”
- 是否存在“专业优先导致学校层次下降过快”

### 3.12 为什么这样设计

这套设计比单纯规则更像工程项目，原因有三点：

- 业务合理：符合高考填报的真实决策逻辑，先保可报，再谈偏好
- 架构清晰：召回、打分、重排、风控可以分别优化
- 面试好讲：能说明为什么不用纯机器学习，以及为什么位次是核心特征

## 4. 工程实现优化

### 4.1 推荐技术栈

如果目标是“简历项目 + 可面试讲解 + 可落地产品”，推荐技术栈如下：

| 层级 | 技术 | 作用 |
| --- | --- | --- |
| 前端 | React + Vite | 构建学生端 / 家长端 / 顾问端 Web 应用 |
| 后端 API | FastAPI | 提供推荐、用户、数据查询、实验接口，适合算法服务化 |
| 数据处理 | Python + Pandas | 清洗录取线、位次表、专业数据，适合批处理和 ETL |
| 数据库 | PostgreSQL | 存储结构化院校、专业、录取事实表和用户数据 |
| 缓存 | Redis | 缓存热门推荐结果、元数据、会话上下文 |
| 定时任务 | Cron / APScheduler | 定时更新数据、生成统计指标、清理缓存 |
| AI 接入 | OpenAI Compatible API | 负责方案解释、二次问答、风险提示生成 |

### 4.2 为什么这样选

#### 前端：React + Vite

- 当前仓库已经是 React，迁移成本最低
- 适合快速做多页面工作台和顾问视图
- 与现有代码一致，延续性最好

#### 后端：FastAPI

- 推荐和数据处理天然更偏 Python 生态
- 适合写特征工程、概率模型、批量数据服务
- 自动生成 OpenAPI 文档，便于前后端联调

#### 数据处理：Pandas

- 录取线、位次表本质是结构化表数据
- 非常适合做清洗、去重、格式标准化、增量对比

#### 数据库：PostgreSQL

- 比 SQLite 更适合生产环境
- 支持复杂查询、索引、JSON 字段、统计分析
- 后续扩展到多省份、多年份更稳

#### 缓存：Redis

- 热门分数段和热门组合查询高度重复
- 推荐结果和位次映射非常适合缓存

### 4.3 如果想和当前仓库最少冲突

可以按两步走：

#### 第一步：保留现有前后端

- 前端继续用 React
- 后端继续用 Express
- 先把推荐模块拆成独立 service
- SQLite 升级为 PostgreSQL
- 增加 Redis 缓存

#### 第二步：把推荐链路迁移到 Python 服务

- Node/Express 作为网关或 BFF
- Python/FastAPI 负责推荐算法和数据分析

这样既保留现有仓库成果，也能体现架构演进能力。

## 5. 性能与工程亮点

### 5.1 高并发查询优化

高考出分后短时间内查询会明显集中，建议增加以下优化：

- 对 `省份 + 年份 + 科类 + 位次` 建立联合索引
- 将省控位次表预加载到内存或 Redis
- 对热门分数段预生成候选集
- 推荐接口拆成“召回缓存 + 排序计算”两段

面试可讲：

> 将高频查询数据从数据库 IO 转为内存 / Redis 查询，降低接口延迟，并把候选召回阶段做成预计算结果，减少重复排序开销。

### 5.2 推荐结果缓存机制

推荐结果很适合做缓存，因为很多用户画像高度相似。

缓存 Key 可设计为：

```text
province:track:score:rank:subjects:risk:preference_hash:data_version
```

缓存策略：

- TTL 30 分钟到 2 小时
- 数据版本变更后自动失效
- 对命中率高的热门档位做主动预热

### 5.3 历史数据增量更新

数据更新不能每次全量覆盖，建议做成：

- `raw`：原始 CSV / PDF 解析结果
- `staging`：字段标准化后的中间表
- `prod`：去重、校验、可检索的正式表

增量逻辑：

- 通过 `province + year + track + university + major` 做唯一键
- 新数据导入前先做 checksum
- 只对变化记录做 upsert
- 导入日志入库，便于追溯

这会让项目从“导文件脚本”升级成“可运维的数据管道”。

### 5.4 用户个性化推荐

可以在用户反馈上做闭环：

- 用户收藏 / 删除某院校专业
- 用户多次强调城市优先或专业优先
- 用户是否接受调剂
- 用户历史咨询主题

然后动态调整权重：

- 城市优先用户，提高 `city_match` 权重
- 专业优先用户，提高 `major_match` 权重
- 保守型用户，提高 `admit_score` 和 `safe tier` 权重

这类个性化策略即使不引入复杂模型，也非常有产品感。

### 5.5 A/B 测试框架

推荐系统非常适合做实验框架。

可对以下策略做 A/B：

- 冲稳保比例
- 保底层厚度
- 专业优先 vs 城市优先权重
- AI 顾问解释模板

实验链路建议：

- 用户进入系统时分配 `experiment_id`
- 请求日志中记录命中的推荐版本
- 埋点收藏率、追问率、导出率、方案停留时长
- 对不同版本效果做对比

面试里这会很加分，因为它说明你考虑的是“推荐效果验证”，而不是只会“推荐结果生成”。

### 5.6 可观测性

建议补充：

- `trace_id` 串联请求链路
- 推荐耗时监控
- 缓存命中率监控
- 数据导入成功率监控
- AI 顾问调用失败率监控

这能体现你有线上产品思维。

## 8. 面试讲解话术

### 8.1 一分钟版本

这个项目本质上是一个智能升学决策系统。它解决的不是简单的“查分换学校”，而是高考志愿填报里信息不对称、滑档风险高和分数利用率低的问题。系统会先根据省份、科类、分数、位次和选科要求召回候选学校，再结合历年录取位次、招生波动和用户的专业/城市偏好做概率估计和排序，最后生成冲稳保三层志愿方案，并通过 AI 顾问解释为什么这样排、哪里有风险、如果用户只想留本省应该怎么改。

### 8.2 三分钟版本

我当时是按推荐系统思路来设计这个项目的。第一层是数据层，核心数据包括省控分数位次表、院校专业历年录取线和选科限制。第二层是推荐层，我把它拆成候选召回、位次匹配、录取概率估计、偏好打分和冲稳保重排。这里位次是核心特征，因为裸分跨年份和跨省份不稳定。第三层是服务层，提供推荐接口、AI 顾问接口、历史记录和后台导入。第四层是展示层，既支持学生自己使用，也支持家长和志愿顾问复盘方案。这个项目的难点主要有两个，一是如何把“能录取”和“用户想去”这两个目标做平衡，二是怎么避免推荐结果看起来很好但保底不够厚，所以我专门设计了整体滑档风险评估和保底层加厚策略。
